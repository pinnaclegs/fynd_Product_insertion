Fynd Store OS Case Study | Part B

Meridian Lifestyle Implementation Plan

A practical eight-week rollout plan for migrating 107 India and UAE stores, 60,000 SKUs, and D2C operations from legacy POS and Shopify onto Store OS.

107 stores 70 apparel, 25 footwear, 12 accessories
60,000 SKUs Catalog migration with validation retry design
8 weeks Festive season deadline
Phased rollout Pilot first, then risk-based waves

Executive Summary

Meridian should be taken live through a controlled phased rollout, not a 107-store big-bang. The plan protects the festive deadline by pre-loading and validating catalog data early, proving store readiness before development completes, and expanding through brand and country-aware waves.

Recommendation Run a pilot, then phased waves by operating model, country compliance, and store readiness.
Critical Path Catalog, inventory, POS invoicing and tax, payments, OMS/D2C, and marketplace feeds.
Primary Risk Catalog defects or hardware readiness gaps discovered too late for festive cutover.
Cutover Principle Pre-load master data before go-live and use overnight windows for deltas and validation.

B1. Implementation Plan and Rollout Sequencing

The lifecycle is intentionally front-loaded with discovery, store readiness, data quality, and critical integration design. This reduces late-stage surprises and keeps the final weeks focused on proving operations at scale.

Eight-Week Delivery Plan

Week Focus What Happens Key Outputs
1 Kickoff, governance, discovery Align scope, owners, success criteria, escalation path, and data requests. Start hardware and store-readiness checks immediately. RAID log, project calendar, scope baseline, data request tracker.
2 Solution design and setup Map store, ecommerce, return, refund, tax, promotion, and inventory flows. Configure sandbox and staging. Integration map, process flows, environment setup, store configuration baseline.
3 Catalog and integration build Run first catalog dry migration. Start POS, inventory, payment, OMS, D2C, and finance integration work. Catalog rejection report, corrected mapping rules, build backlog.
4 End-to-end testing Validate sale, invoice, payment, inventory decrement, online reservation, fulfillment, return, refund, and finance export. E2E test evidence, defect list, performance baseline.
5 UAT and training Business users and pilot stores test real scenarios. Train super users, area managers, and store leads. UAT signoff, SOPs, training attendance, go-live checklist.
6 Pilot go-live Launch a risk-representative pilot across brands, store types, and geography. Run daily incident triage. Pilot stability report, fixes, wave readiness decision.
7 Scale rollout Expand through apparel waves and prepare footwear/accessories using lessons from pilot. Wave signoffs, delta migration reports, support readiness.
8 Final rollout and hypercare Complete remaining stores, switch critical channels, reconcile data, and move into festive hypercare. Go-live completion, reconciliation, hypercare dashboard.

Rollout Sequencing

I would avoid a 107-store big-bang. The safer plan is a fast phased rollout with a representative pilot, then brand and country-aware waves.

Wave Stores Scope Boundary Logic
Pilot 7 4 apparel, 2 footwear, 1 accessories, including varied volume, format, hardware, and geography. Expose real operating differences early rather than picking only easy stores.
Wave 1 30-35 Priority apparel stores. Largest revenue footprint and similar catalog/workflow model.
Wave 2 35-40 Remaining apparel stores. Scale after 2-3 stable trading days from Wave 1.
Wave 3 25 Footwear stores. Different size curves, returns behavior, and category logic.
Wave 4 12 Accessories stores. Smaller footprint and lower complexity, unless business priority changes.
Wave principle: cut by operational similarity, country compliance, store readiness, hardware profile, payment provider, and revenue risk. If UAE VAT or payment behavior differs materially, India and UAE should be separate sub-waves.

B2. Sizing and Analytical Reasoning

This section answers one practical question: can Meridian safely load 60,000 SKUs during an overnight cutover? The API time itself is manageable, but the real risk is the time needed to correct failed records and reconcile the load.

Ingestion Calculations

Input Value Why It Matters
Total catalog 60,000 SKUs Every SKU must be created or updated on Store OS.
API speed 5 requests per second Each request processes one product, so this controls the fastest possible load time.
First-pass failures 8% Failed records need correction and one more API attempt.
Cutover window About 8 hours The full load, corrections, retry, and reconciliation must fit overnight.

1. Clean Single-Threaded Load

60,000 SKUs / 5 requests per second = 12,000 seconds 12,000 seconds / 60 = 200 minutes 200 minutes / 60 = 3.33 hours Clean load time = about 3 hours 20 minutes

If every record is clean and the job runs continuously at 5 requests per second, the first pass takes approximately 3 hours 20 minutes.

2. Add the 8% Failure Retry

Failed records = 60,000 x 8% = 4,800 SKUs Retry time = 4,800 / 5 requests per second = 960 seconds 960 seconds / 60 = 16 minutes API time including retry = 3 hours 20 minutes + 16 minutes = 3 hours 36 minutes
Assumption Used in Estimate
The API sustains 5 requests per second for the full run. Yes
Each request creates or updates one SKU. Yes
8% fail only once, are corrected, and pass on the second attempt. Yes
Correction time is separate from API retry time. Yes
No major API outage, network issue, or rate-limit slowdown occurs. Yes

Realistic End-to-End Estimate

Step Time Simple Explanation
Initial load 3h 20m Send all 60,000 SKUs once.
Retry failed 8% 16m Send the 4,800 corrected SKUs again.
Validation report and correction 1h-2h Review failed records, fix data, and prepare retry file.
Reconciliation and buffer 1h-2h Confirm counts, spot-check products, handle transient errors.
Total realistic estimate 5.5h-7.5h Fits only if the catalog has already been dry-run before cutover.
Cutover answer: the estimate can fit inside eight hours, but only with prior dry runs. I would not do the first full validation during the final overnight cutover.

Options If the Window Does Not Fit

Change Why I Would Do It Trade-Off / Risk
Pre-load the catalog before cutover Move the biggest job out of the overnight window and use cutover for final deltas. Requires catalog freeze or strong delta tracking.
Run dry migrations earlier Find and fix bad barcodes, tax codes, categories, and mandatory fields before go-live night. Needs committed Meridian catalog owners.
Use approved bulk or parallel ingestion Shorten the API processing time if Store OS allows it. May hit platform limits or downstream load if not tested.
Split by brand Load apparel, footwear, and accessories in line with rollout waves. Needs clean brand isolation across D2C and marketplaces.
Use idempotent upsert Safely load the same SKU again without creating duplicates. Requires reconciliation of price, stock, and active status at cutover.

Crash-Resumable Ingestion Design

The ingestion job should behave like a checklist. Every SKU gets a stable external ID, such as legacy SKU ID or brand plus barcode. The job stores the status of each SKU in an ingestion ledger.

Ledger Field Purpose
SKU external ID Prevents duplicates by identifying the same product every time.
Source record hash Shows whether the source data changed since the last successful load.
Status Tracks pending, processing, success, failed, or skipped.
Target product ID Stores the Store OS product ID after successful create or update.
Attempt count and last error Helps retry failed records and report correction reasons.
If the job crashes at 70%: restart it and process only records where status is not success, or where the source hash has changed. Successful unchanged SKUs are skipped, so the job does not create duplicates or redo completed work.

B3. Architecture Articulation

In plain terms, Meridian's Store OS architecture has one central operating layer that connects stores, online channels, payments, catalog, inventory, and the merchant's existing finance or ERP systems. The goal is simple: every channel sells from the same trusted catalog and the same real-time view of available stock.

Major Components and Data Flow

Component Role in the Implementation How It Connects
Catalog Stores SKU, barcode, brand, size, color, category, price, tax, image, and channel eligibility data. Loaded from legacy/Shopify/ERP through the integration layer, then published to POS, D2C, and marketplaces.
POS Runs store billing, returns, exchanges, invoices, and store user workflows. Uses Store OS catalog, price, tax, payment, and inventory services in real time.
OMS Receives online and marketplace orders, reserves stock, routes fulfillment, and handles cancellations/returns. Consumes channel orders and updates inventory reservations and fulfillment status.
Inventory Maintains sellable, reserved, damaged, returned, and in-transit stock by store and warehouse. Receives stock movements from POS, OMS, warehouse, and returns; publishes available stock to online channels.
Merchant systems Legacy POS, ERP/finance, WMS, and Shopify provide starting data and may remain systems of record for finance or planning. Connected through APIs, file imports, webhooks, or scheduled exports depending on system capability.
Payments Handles card, UPI, wallet, cash, refunds, and reconciliation flows. POS initiates payment and stores transaction references for invoice and finance reconciliation.
D2C / marketplaces Customer-facing channels for online orders and product discovery. Receive catalog and stock availability from Store OS; send orders back into OMS.

Inventory Source of Truth

After go-live, Store OS inventory should be the operational source of truth for sellable stock. ERP can remain the financial or planning system, but Store OS should be the system that decides whether an item is available to sell in a store, on D2C, or on marketplaces.

Event Inventory Behavior Channel Sync
Store sale POS creates invoice and Store OS decrements stock for that store. Updated availability is pushed to D2C and marketplaces.
Online order OMS reserves stock immediately so stores and other channels cannot oversell it. Available-to-promise reduces across online and offline channels.
Cancellation Reserved stock is released back to sellable inventory. Availability is republished to eligible channels.
Return Stock is added back only after return acceptance and quality status. Sellable quantity updates only if the returned item is approved for resale.
Warehouse or store transfer Stock moves through in-transit status until received at destination. Only received sellable stock becomes available to channels.

Expected Load Hotspot

The most likely scale issue at 107 stores is inventory event latency during peak festive trading. Store invoices, online orders, returns, cancellations, and marketplace reservations can all hit inventory at the same time. If processing falls behind, online and offline stock can drift apart.

Risk What I Would Monitor Alert Threshold Example
Inventory updates lag behind store billing and online reservations. Inventory queue depth, p95/p99 processing latency, failed events, dead-letter count, API rate-limit responses, stock sync lag by channel, oversell/cancellation rate. Warning if p95 inventory latency exceeds 60 seconds; critical if it exceeds 180 seconds or failed inventory events exceed 0.5% over 15 minutes.
Client-facing explanation: Store OS keeps stock consistent by treating every sale, reservation, cancellation, return, and transfer as an inventory event. Engineering-facing explanation: the inventory event pipeline must be observable, retryable, idempotent, and protected by alerts before the festive rollout.

B5. Go-Live Day Scenario

At 11 PM, invoices are generating but inventory is not decrementing for one product category. Forty stores are scheduled to open at 6 AM. I would treat this as a go-live incident with a hard decision point no later than 3:30-4:00 AM.

Incident Timeline

Timeframe What We Do What We Need to Know
First 15 minutes Start the incident bridge, pause non-essential changes, and get real examples from stores. Which stores, SKUs, invoices, and timestamps are affected?
Next 45 minutes Compare the failing product category with a category that is working correctly. Is the issue caused by product setup, category mapping, stock rules, or inventory event processing?
By 2:00 AM Measure the business impact and decide whether a temporary workaround is practical. How important is this category, and can stores safely sell without automatic stock decrement?
By 3:30-4:00 AM Make the go/no-go decision with Meridian's sponsor and Store OS leadership. Do we proceed, proceed with restrictions, or keep stores on the fallback plan?
Before 6:00 AM Send one clear instruction to all stores and correct any stock already impacted. Every store knows exactly which system to use and what to do at opening.
Simple rule: by 4:00 AM, stores should have one clear instruction: go live as planned, go live with a controlled restriction, or stay on the fallback system for the day.

Decision Matrix

Condition Decision Rationale
Root cause is identified, fixed, and regression-tested across sale, return, inventory, and online sync. Proceed Operational risk is controlled before store opening.
Issue is isolated to one non-material category and can be blocked or manually controlled. Partial proceed Business can trade with explicit workaround and reconciliation plan.
Root cause is unknown, category is material, or inventory decrement is unreliable beyond one category. Delay / rollback Opening 40 stores with unreliable inventory creates oversell and reconciliation risk.
Likely decision: if the issue is truly category-specific and a tested fix or controlled block is ready before the decision checkpoint, proceed partially with written business signoff. If root cause remains unknown by cutoff, delay the wave and keep stores on legacy for that trading day.
